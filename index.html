Mozilla IAM API Reference

IAM is an authorization micro-service that allows to checks if an arbitrary subject is allowed to perform an action on a resource, based on a set of rules (policies).

It is inspired by AWS IAM Policies.

Version: 0.1

Doorman

Main API endpoints.

Doorman

Check authorization request

POST /allowed

Are those principals allowed to perform this action on this resource in this context?

Optional authentication

When the service runs with the JWT_ISSUER environment variable, the authentication is mandatory.

With the JWT_ISSUER configured, the principals cannot be specified in the authorization request. They are taken from the JWT payload (userid:{Subject}, email:{Email}, group:{Groups..}, etc.)

Authorization request in JSON.

Note that every field is optional.

principals: string[]

Arbitrary list of strings (eg. userid:alice). Only if the server runs without authentication

action: string

Any action (eg. read, delete, signoff)

resource: string

Any resource (eg. blocklist, /update/rules/35)

context: object

Any extra information to be matched in policies conditions. The context fields remoteIP and audience will be forced by the server. The values provided in the roles context field will expand the principals with extra role:{} values.

Origin

The JWT claimed audience will be checked against this Origin request header (eg. https://api.service.org).

The specified value must also match one of the known audience from the policies files.

type
string
in
header
Authorization

With the JWT_ISSUER configured, a valid JSON Web Token (JWT) must be provided in the Authorization request header. (eg. Bearer eyJ0eXAiOiJKV1QiLCJhbG...9USXpOalEzUXpV)

type
string
in
header
Request Content-Types: application/json
Request Example
{
  "principals": [
    "userid:ldap|ada",
    "email:ada@lau.co"
  ],
  "action": "create",
  "resource": "comment",
  "context": {
    "env": [
      "stage"
    ],
    "roles": [
      "changer"
    ]
  }
}
200 OK

Return whether it is allowed or not.

400 Bad Request

Missing headers or invalid posted data.

401 Unauthorized

JWT token is invalid.

403 Forbidden

JWT claimed audience is invalid (authorization failed).

Response Content-Types: application/json
Response Example (200 OK) 
{
  "allowed": true,
  "principals": [
    "userid:ldap|ada",
    "email:ada@lau.co",
    "tag:mayor",
    "role:changer"
  ]
}
Response Example (400 Bad Request) 
{
  "message": "Missing `Origin` request header"
}
Doorman

Reload the policies

POST /__reload__

Reload the policies (synchronously). This endpoint is meant to be used as a Web hook when policies files were changed.

It would be wise to limit the access to this endpoint (e.g. by IP on reverse proxy)

200 OK

Reloaded successfully.

500 Internal Server Error

Reload failed.

Response Content-Types: application/json
Response Example (200 OK) 
{
  "success": true
}
Response Example (500 Internal Server Error) 
{
  "success": false,
  "message": "could not parse YAML in \"https://github.com/ops/conf/policies.yaml\""
}
